ReST API

By Deepti Menon

Table of Contents

In modern web development, APIs (Application Programming Interfaces) serve as the backbone for communication between software components. Among them, REST (Representational State Transfer) APIs are the most popular due to their simplicity, scalability, and stateless communication style.

What is a REST API?

A REST API is an interface that allows applications to communicate over the internet by following certain constraints and principles. It uses HTTP methods to perform operations on resources, enabling scalable, stateless, and decoupled systems.

REST API Components

Request

  • URI: Specifies the resource we want to access.
  • Methods: Common HTTP methods include:
    • GET – Fetch a resource
    • POST – Create a new resource
    • PUT – Update an existing resource
    • DELETE – Remove a resource
  • Headers: Metadata such as:
    • Auth – Authentication token
    • Content-Type, Cookie, Cache-Control
  • Body: For POST and PUT, contains data to send to the server.

Response

  • Status Line: Includes codes like:
    • 200 OK, 201 Created, 400 Bad Request
    • 401 Unauthorized, 404 Not Found, 500 Internal Server Error
  • Headers: Info like content type, token, etc.
  • Body: Actual data or message returned.

REST API Workflow

  1. Client Initiates Request
    • Sends a request to a specific API URI.
    • Includes HTTP method, headers, and optionally a body.
  2. Authentication & Authorization Check
    • Server validates client identity using:
      • API Key
      • JWT/OAuth Token
      • Basic Authentication
    • If authentication fails, server responds with:
      • 401 Unauthorized — when no valid credentials are provided
      • 403 Forbidden — when access to the resource is not permitted
  3. Request Routing & Processing
    • Server routes the request to the appropriate controller or handler based on URI and method.
    • Validates request parameters and body.
  4. Business Logic Execution & Data Access
    • Server interacts with database or services to fetch, create, update, or delete data as needed.
    • Applies any business logic, validations, or transformations.
  5. Response Creation
    • Server builds the response with:
      • HTTP status code (200, 201, 400, 500)
      • Headers (e.g., Content-Type, Cache-Control, and updated Auth tokens if needed)
      • Response body (JSON/XML data or error message)
  6. Response Sent to Client
    • Response is delivered back to the client application.
    • Client processes the response to update UI, store tokens, or handle errors.

Benefits of REST APIs

  • Scalability: Statelessness allows APIs to scale easily across multiple servers without session management complexities.
  • Flexibility & Decoupling: Client and server operate independently. The client only requires the API contract (URI and data structure) to interact with the server.
  • Technology Independence: Any language, platform, or framework can be used on either end as long as communication happens via HTTP.
  • Ease of Caching: Responses can be cached at the client or intermediary servers based on headers, reducing redundant requests and boosting speed.
  • Improved Maintainability: Clear separation between frontend and backend development makes APIs easier to debug, update, and scale.
  • Performance Optimization: APIs can serve data in client-preferred formats (JSON, XML) and optimize endpoints for different devices like web, mobile, or IoT.

Core REST Principles

  • Statelessness: Each request carries all the required data for the server to fulfill it. No session or state is stored on the server between client requests.
  • Uniform Interface: APIs follow a consistent way of interacting with resources using standard URIs and HTTP methods like GET, POST, PUT, and DELETE.
  • Layered System: The API architecture can include multiple intermediary servers such as proxies, load balancers, or firewalls. Clients remain unaware of these intermediaries.
  • Cacheability: Responses should indicate if they’re cacheable, allowing clients and proxies to store responses for subsequent requests, improving efficiency and reducing server load.
  • Code on Demand (Optional): Servers can enhance client functionality by sending executable code like JavaScript. This principle is optional and rarely used in most REST APIs.

Detailed Authentication in REST APIs

Authentication verifies the identity of a client attempting to access an API. Since REST APIs are stateless, each request must independently carry credentials for authentication.

Common Authentication Methods

API Key Authentication

A unique key is issued to a client and passed with every request, usually in a header or query parameter.

GET /api/customers HTTP/1.1 Authorization: ApiKey 12345xyz
  • Simple to implement
  • Not very secure if not used over HTTPS
  • Hard to revoke or track individual keys

Basic Authentication

Sends a Base64-encoded string of username:password in the header.

Authorization: Basic dXNlcjpwYXNzd29yZA==
  • Very simple to set up
  • Unsafe unless over HTTPS
  • Sends credentials with every request

Bearer Token (JWT/OAuth Token)

The client sends a token with each request in the Authorization header. The server validates this token before processing the request.

Authorization: Bearer eyJhbGciOiJIUzI1...
  • Secure and stateless
  • Supports expiration, revocation, and embedded claims
  • Can contain user roles and permissions

OAuth 2.0 Authentication

An industry-standard framework for secure, delegated API access, especially popular with large platforms like Google and Facebook.

  • Highly secure and scalable
  • Allows third-party apps to act on behalf of users without sharing passwords
  • More complex to set up

Authentication in the Workflow

  1. The client sends a request with credentials (API key, token, etc.).
  2. The server intercepts and validates credentials before routing or business logic execution.
  3. If invalid, the server immediately responds with 401 Unauthorized or 403 Forbidden.
  4. If valid, the request proceeds to processing, database access, and response generation.

Why is Authentication Critical?

  • Prevents unauthorized access to services and data.
  • Secures sensitive transactions and operations.
  • Enables usage monitoring and logging per user or application.
  • Protects APIs from abuse and malicious activity.

HTTP Methods Explained (In-Depth)

HTTP methods are verbs that define the action to be performed on a resource in a REST API. They are central to REST’s uniform interface constraint and guide how clients interact with resources.

GET

  • Purpose: Retrieve data from the server.
  • Request Body: Not allowed.
  • Response: Data in JSON, XML, or plain text format.
GET /api/customers/123

POST

  • Purpose: Create a new resource.
  • Request Body: Contains the new resource data.
  • Response: 201 Created with the new resource URI or data.

POST /api/customers
Body: {
    "name": "Alex",
    "email": "alex@example.com"
}

PUT

  • Purpose: Update an existing resource or create it if absent.
  • Request Body: Full updated resource data.
  • Response: 200 OK or 201 Created.

PUT /api/customers/123
Body: {
    "name": "Alex Updated",
    "email": "alex@example.com"
}

PATCH

  • Purpose: Partially update a resource.
  • Request Body: Only the fields to be updated.
  • Response: 200 OK or 204 No Content.

PATCH /api/customers/123
Body: {
    "email": "alex.updated@example.com"
}

DELETE

  • Purpose: Remove a resource.
  • Request Body: Not typically required.
  • Response: 200 OK or 204 No Content.
DELETE /api/customers/123

Conclusion

REST APIs are the cornerstone of modern web application architecture. By understanding how they work and following REST principles, developers can build efficient, secure, and scalable systems.
