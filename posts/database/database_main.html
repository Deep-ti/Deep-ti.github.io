Database

By Deepti Menon

Table of Contents

A database is an organized collection of data, typically stored and accessed electronically from a computer system.

A Database Management System (DBMS) is the software that allows users and applications to interact with the database.

A Query Language is used by databases to communicate with the DBMS. The most common is SQL (Structured Query Language), primarily used for relational databases.

Structures of Database

Relational Databases (SQL Databases): Data is organized into tables, which consist of rows (records) and columns (attributes).
Examples: MySQL, PostgreSQL, Oracle, Microsoft SQL Server.
Use Cases: E-commerce platforms, banking systems, inventory management, CRM systems, where data relationships and integrity are crucial.

Object Databases: Data is stored in the form of objects, as they are used in object-oriented programming (OOP) languages.
Examples: Objectivity/DB, GemStone/S, ObjectDB, ObjectDatabase++.
Use Cases: CAD/CAM (Computer-Aided Design/Manufacturing), GIS (Geographic Information Systems), Multimedia Applications, Telecommunications, Real-time systems.

Object-Relational Databases: Essentially a relational database extended with features that support the object-oriented paradigm directly within its schema and query language.
Examples: PostgreSQL, Oracle, IBM DB2.
Use Cases: E-commerce platforms, banking systems, inventory management, CRM systems, CAD/CAM (Computer-Aided Design/Manufacturing), GIS (Geographic Information Systems), Multimedia Applications, Telecommunications, Real-time systems (these overlap as ORDBs combine features).

Non-Relational Databases (NoSQL Databases): Data is organized in unstructured or semi-structured formats. They scale horizontally better than traditional relational databases for certain use cases.
Types include:

  • Document Databases: Store data in flexible, document-like formats (e.g., JSON). (e.g., MongoDB, Couchbase)
  • Key-Value Stores: Store data as simple key-value pairs. (e.g., Redis, DynamoDB)
  • Column-Family Stores: Store data in columns that are grouped together. (e.g., Cassandra, HBase)
  • Graph Databases: Store data as nodes and edges to represent relationships. (e.g., Neo4j)

Use Cases: Social media applications, real-time analytics, content management systems, applications with large volumes of rapidly changing data.

PostgreSQL

PostgreSQL is an open-source descendant of the original Berkeley code. It supports a large part of the SQL standard and offers many modern features:

  • complex queries
  • foreign keys
  • triggers
  • updatable views
  • transactional integrity
  • multiversion concurrency control (MVCC)
Also, PostgreSQL can be extended by the user in many ways, for example by adding new
  • data types
  • functions
  • operators
  • aggregate functions
  • index methods
  • procedural languages

Advanced Features

  • Views: Views allow you to encapsulate the details of the structure of your tables, which might change as your application evolves, behind consistent interfaces.
  • Foreign Keys: A foreign key constraint specifies that the values in a column (or a group of columns) must match the values appearing in some row of another table.
    Note:
    • A foreign key can also constrain and reference a group of columns.
    • Self-referential foreign key: the "other table" of a foreign key constraint can be the same table.
    • A table can have more than one foreign key constraint. This is used to implement many-to-many relationships between tables.
    • Delete options:
      • RESTRICT: prevents deletion of a referenced row.
      • NO ACTION: if any referencing rows still exist when the constraint is checked, an error is raised.
      • CASCADE: when a referenced row is deleted, row(s) referencing it should be automatically deleted as well.
      • SET NULL: referencing column(s) in the referencing row(s) to be set to nulls.
      • SET DEFAULT: referencing column(s) in the referencing row(s) to be set to defaults.
  • Transactions: A transaction is set up by surrounding the SQL commands of the transaction with BEGIN and COMMIT commands.
    After defining a savepoint with SAVEPOINT, you can if needed roll back to the savepoint with ROLLBACK TO. All the transaction's database changes between defining the savepoint and rolling back to it are discarded, but changes earlier than the savepoint are kept. After rolling back to a savepoint, it continues to be defined, so you can roll back to it several times.   • Window Functions: A window function performs a calculation across a set of table rows that are somehow related to the current row. This is comparable to the type of calculation that can be done with an aggregate function. However, window functions do not cause rows to become grouped into a single output row like non-window aggregate calls would. Instead, the rows retain their separate identities. 
    SELECT depname, empno, salary, avg(salary) OVER (PARTITION BY depname) FROM empsalary;
  • Inheritance: 
    CREATE TABLE capitals (
    state        char(2) UNIQUE NOT NULL
  ) INHERITS (cities);
    In this case, a row of capitals inherits all columns (name, population, and elevation) from its parent, cities. ONLY before cities indicates that the query should be run over only the cities table, and not tables below cities in the inheritance hierarchy. Many of the commands that we have already discussed — SELECT, UPDATE, and DELETE — support this ONLY notation.
